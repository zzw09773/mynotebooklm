流程骨架是一樣的，但每個步驟的決策邏輯會隨主題變化。我拆開來講每個階段我在想什麼。

**一、判斷工作模式**

收到請求後我先判斷三件事：

有沒有提供模板 .pptx？如果有，我走 `editing.md` 的流程（解包 XML → 替換內容 → 重新打包），保留原本的設計風格。如果沒有，就走 `pptxgenjs.md` 從零建立。

有沒有提供來源素材？可能是 PDF、Word、CSV、一段文字、或只有一個主題。素材量決定了我需要「整理濃縮」還是「擴展發想」。

簡報的用途是什麼？匯報型（像你這次的心得報告）、提案型（pitch deck）、教學型、還是資訊展示型，版面風格完全不同。

**二、內容規劃**

這是最關鍵的一步，跟主題高度相關。我的思考方式是：

先決定「敘事弧線」。匯報型通常是「背景 → 做了什麼 → 成果 → 下一步」。提案型是「痛點 → 方案 → 為什麼是我們 → CTA」。教學型是「概念 → 原理 → 範例 → 練習」。不同弧線決定了投影片的排列順序。

然後決定每頁的「一個核心訊息」。好的投影片是一頁一個重點，不是把所有資訊塞在一起。如果一頁想講兩件事，就拆成兩頁。

最後決定每頁適合的版面類型。我常用的大概有這些：

| 版面類型 | 適合的內容 |
|----------|-----------|
| 大數字指標卡 | KPI、統計數據、成果展示 |
| 左右雙卡對照 | 比較、優劣分析、Before/After |
| 多格卡片（3~6 格） | 分類、功能列表、團隊介紹 |
| 表格 | 規格對比、時程表、數據清單 |
| 流程圖（圓形+箭頭） | 步驟、階段、生命週期 |
| 半圖半文 | 產品展示、場景說明 |
| 全版深色+大字 | 金句、章節轉場、CTA |
| 圖表（長條/圓餅/折線） | 趨勢、佔比、量化數據 |

我會刻意讓相鄰的投影片不重複同一種版面，避免視覺疲勞。

**三、設計決策**

配色方面，SKILL.md 提供了 10 組預設色票，但我不會直接套用，而是根據主題選擇。例如醫療健康主題我可能選 Teal Trust 或 Sage Calm，金融主題選 Midnight Executive，食品或生活主題選 Warm Terracotta 或 Coral Energy。核心原則是一個主色佔 60~70%，一個輔色，一個強調色。

字型方面，正式場合我偏好 Georgia + Calibri 這種襯線標題配無襯線內文的組合。科技主題可能用 Consolas + Calibri。如果是中文為主的簡報，Georgia 處理中文也還行，因為系統會 fallback 到內建中文字型。

圖示方面，react-icons 有幾千個圖示可用，我會根據主題挑選對應的 icon set。商業主題用 Font Awesome 的圖表、金錢、人物圖示，技術主題用 Material Design 的 code、server、cloud 圖示。

**四、寫程式碼的策略**

我不是從第一行寫到最後一行。實際的撰寫順序是：

首先定義全域常數，包括色票、字型、共用的 shadow 工廠函式。這些會被所有投影片重複使用，先定好可以確保一致性。

然後做封面和結尾頁，因為這兩頁定義了整份簡報的視覺調性。

接著做內容最複雜的那一頁（通常是表格或多格卡片），確認座標計算邏輯正確。

最後依序填完其餘頁面。很多頁面的結構類似（標題 + 裝飾線 + 內容區），我會建立一致的模式：標題在 y=0.4，裝飾線在 y=0.95，內容從 y=1.2 開始。

**五、沒有素材時怎麼辦**

如果你只給我一個主題，比如「幫我做一份關於永續能源的簡報」，我會先問你幾個問題來縮小範圍：對象是誰、目的是什麼、大概要幾頁、有沒有特別想強調的數據或論點。如果你說「你自己決定就好」，我就會根據主題用我的知識組織內容，產出一份結構完整的簡報，但會提醒你檢查數據的正確性。

**六、有模板時的不同做法**

如果你給我一個 .pptx 模板，流程會變成：

```bash
# 先看模板長什麼樣
python scripts/thumbnail.py template.pptx

# 解包成 XML
python scripts/office/unpack.py template.pptx unpacked/

# 分析版面配置（母片、佔位符）
# 修改 XML 中的文字、圖片
# 清理多餘元素
python scripts/clean.py unpacked/

# 重新打包
python scripts/office/unpack.py --pack unpacked/ output.pptx
```

這樣可以完美保留原模板的設計風格，只替換內容。

**總結來說，不管什麼主題，核心邏輯就是四個字：先想再做。** 先想清楚敘事弧線和每頁的核心訊息，再選擇對應的版面和視覺風格，最後才進入程式碼撰寫。程式碼本身只是把設計意圖翻譯成座標和 API 呼叫，真正決定簡報品質的是前面的規劃階段。
